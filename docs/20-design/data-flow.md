# 設計：データ取得の流れ

「どこから・どうやって取得し・どう保存し・どう表示するか」を整理する。
※ 下記のAPIは実際に叩いて動作確認済み。

## 1. 全体像（データの流れ）

```
 ① 日本人選手を自動抽出     ② MLB公式API            ③ 整形・保存          ④ 表示
┌──────────────┐          ┌──────────────┐  JSON   ┌──────────────┐      ┌──────────────┐
│ 在籍選手一覧から │ ──────▶ │ statsapi.mlb │ ──────▶ │ data/        │ ───▶ │ ダッシュボード │
│ birthCountry= │  各選手  │ .com         │         │ latest.json  │      │ (Webページ)   │
│ Japan を抽出   │  のID    │              │         │ history/     │      └──────────────┘
└──────────────┘          └──────────────┘         │  2026-05-31  │            ▲
                                                    └──────────────┘            │
                                                          │ 日付ごとに溜まる ─────┘
```

- **更新ボタン** を押すと ①→②→③ が走り、日本人選手を自動抽出して最新データを取得・保存し直す。
- ページ表示（④）は保存済みの `latest.json` を読むだけなので速い。

## 2. データの取得元（MLB公式 StatsAPI）

- ベースURL：`https://statsapi.mlb.com/api/v1`
- **無料・申請不要・APIキー不要**。
- 主に使うエンドポイント（すべて動作確認済み）：

| 目的 | エンドポイント例 | 取得できるもの |
|------|----------------|--------------|
| 選手ID検索 | `/people/search?names=Yoshinobu Yamamoto` | 名前→選手ID（初回の紐付け用） |
| 在籍選手一覧 | `/sports/1/players?season=2026` | 全選手＋`birthCountry`（日本人抽出に利用可） |
| 選手の基本情報 | `/people/{id}` | 氏名・ポジション・所属チーム |
| シーズン通算（打撃） | `/people/{id}/stats?stats=season&group=hitting&season=2026` | 打率・本塁打・打点・OPS 等 |
| シーズン通算（投球） | `/people/{id}/stats?stats=season&group=pitching&season=2026` | 防御率・勝敗・奪三振・WHIP 等 |
| 試合ごとの成績 | `/people/{id}/stats?stats=gameLog&group=hitting&season=2026` | 1試合ごとの成績（昨日の試合を抽出） |

## 3. 対象選手の決め方（自動抽出）

- **固定の選手リストは持たない。** その時点でMLBに在籍している日本人選手を毎回自動で拾う。
- 手順：
  1. `/sports/1/players?season=2026` で在籍選手一覧を取得。
  2. `birthCountry == "Japan"` の選手だけに絞り込む（＝日本人選手）。
  3. 各選手の `id`・`fullName`・ポジションをそのまま使う。
- これにより、移籍・昇格・引退・新規メジャー昇格があっても手直し不要。
- 実データ確認（2026年6月時点）：在籍約1,162人中、日本出身は14人を正しく抽出。

### 日本語名の対応について

- MLBデータの選手名は英語（例：`Munetaka Murakami`）。表示は日本語にしたいので、
  英語名→日本語名の **対応表** だけを小さな設定ファイル（`name_map.json`）で持つ。
- 対応表に無い新規選手が現れた場合は、英語名のまま表示し、後から対応表に足せばよい（成績取得には影響しない）。

## 4. 取得〜整形〜保存の手順（更新ボタンを押したときの処理）

1. 在籍選手一覧から `birthCountry == "Japan"` の選手を抽出（→ 対象選手リストを自動生成）。
2. `name_map.json` で各選手の日本語名を引く（無ければ英語名のまま）。
3. 各選手について：
   - 基本情報 `/people/{id}` → 所属チーム・ポジションを取得。
   - 区分（投手/野手/二刀流）に応じて、シーズン通算と試合ログを取得。
   - 試合ログから「昨日（米国時間の前日）」の試合を抽出。なければ「試合なし」。
4. **今シーズンの成績が全く無い選手は除外する**（例：故障者リスト中で未出場・未登板）。
5. 取得結果を、表示しやすい形に整形（投手項目・野手項目に振り分け）。
6. 保存：
   - `data/latest.json` … 最新の表示用データ（ページが読むのはこれ）。
   - `data/history/2026-05-31.json` … その日のスナップショットを残す（US7：溜めていく）。
7. ページを再読み込み（または自動反映）して最新表示。

## 5. 構成（どう動かすか）

「開発した感」と「将来の自動更新」を両立するため、**小さなローカルのプログラム（バックエンド）＋Webページ（フロント）** の構成にする。

- **バックエンド**：MLB APIからデータを取得・整形・保存し、ページに渡す役割。
- **フロント**：保存されたデータを見やすく表示するダッシュボード。「更新」ボタンでバックエンドに取得を依頼。
- **自動更新（将来）**：毎朝決まった時刻にバックエンドの取得処理を自動で走らせる仕組みを追加するだけでよい（フロントは変更不要）。

※ 具体的なプログラミング言語・ライブラリの選定は、次の「実装設計」で確定する（このドキュメントでは仕組みの流れまで）。

## 6. 注意点

- **時差**：MLBは米国時間で記録されるが、表示は**日本時間に変換**している。各試合の開始時刻を `/schedule?gamePks=...` から取得し、+9時間して日付を出している（例：米国 5-31 の試合 → 日本 6-1）。
- **取得失敗時**：APIが一時的に応答しない場合は、前回保存した `latest.json` をそのまま表示し、エラーメッセージを添える。
- **今シーズン成績が無い選手**：マイナー所属・昇格前・故障者リスト中などで今季成績が空の選手は **非表示**（成績が出れば自動で表示される）。
- **負荷配慮**：選手12人×数リクエスト程度なので軽量。短時間に何度も連打しないようにする。
